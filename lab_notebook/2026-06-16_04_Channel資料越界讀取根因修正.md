# 電子研究紀錄簿

## 標題
`detect.py` Channel 資料越界讀取根因修正（跨通道資料污染解決）

## 日期
2026-06-16

## 研究目的 / 背景
承接前一筆紀錄之診斷結果：`std.001-after flatten` 各 channel 的
`Data length` 標頭欄位皆顯示僅 `25088 bytes`（= 49 行有效資料，相對
標頭宣告的 256 行僅 19%），但 `read_nanoscope_channel()` 目前完全未
參照此欄位，永遠依標頭宣告的 `n_px × n_lines × bpp`（即假設跑滿
256 行）讀取資料，造成讀取範圍跨越到下一個 channel 的資料區。
本次任務為確認此假設並完成修正，解決 `std.001` 推論結果出現
`121.5×` 過度去卷積假警報的根本原因。

## 方法 / 過程

### 1. 跨 channel 偏移量交叉驗證
比對 `inspect_header.py` 輸出的各 channel `Data offset`：

| Channel | Data offset | Bytes/pixel | Data length |
|---|---|---|---|
| Height Sensor | 80960 | 2 | 25088 |
| DMTModulus | 131136 | 2 | 25088 |
| Adhesion | 181312 | 2 | 25088 |
| Deformation | 206400 | 2 | 25088 |
| Dissipation | 231488 | 2 | 25088 |
| Height | 256576 | 2 | 25088 |

計算 `Height Sensor` 與下一 channel `DMTModulus` 的 offset 差距：

```
131136 − 80960 = 50176 bytes
```

但 `read_nanoscope_channel()` 對 `Height Sensor` 實際請求讀取的位元組數：

```
n_px × n_lines × bpp = 256 × 256 × 2 = 131072 bytes
```

`131072 bytes` 遠大於通道間隔 `50176 bytes`，且更遠大於該 channel
自身宣告的有效資料量 `25088 bytes`。讀取範圍 `[80960, 80960+131072)
= [80960, 212032)` 完全覆蓋了 `DMTModulus`（80960~131136 起算）、
`Adhesion`（181312 起算）、`Deformation`（206400 起算）等後續多個
channel 的標頭與資料區，證實「跨 channel 資料污染」假設成立。

### 2. 確認 std 異常值來源
`Height Sensor` 原始讀取統計 `raw min/max = -32768 / 18330`、
`std = 6830.00`。`-32768` 恰為 16-bit 有號整數的最小值，符合「讀到
不相干資料且夾帶異常邊界值」的特徵，與真實高度感測訊號的連續分布
特性不符，進一步支持污染假設。

### 3. 修正設計
**`parse_nanoscope_header()`**：新增解析每個 channel 區塊中的
`\Data length:\s*(\d+)` 欄位，存入 `channels[i]['data_length']`。

**`read_nanoscope_channel()`**：讀取位元組數改為

```python
read_bytes = n_px * n_lines * bpp
if ch.get('data_length'):
    read_bytes = min(read_bytes, ch['data_length'])
```

即以「標頭宣告全幅大小」與「該 channel 實際宣告的有效資料量」兩者
**取較小值**作為讀取邊界，確保不會讀到下一個 channel 的資料區。
既有的「依實際讀到的元素數反推 `actual_lines`，再 zoom 縮放至
128×128」邏輯維持不變，可正確接手處理此邊界值（此次案例
`actual_lines = 25088/2/256 = 49` 行）。

### 4. 工具同步更新
`inspect_header.py` 新增「未限制長度」與「以 Data length 為界」兩種
讀法的並列輸出（min/max/std 對照），作為日後驗證修正效果、排查類似
部分掃描問題的標準診斷流程。

## 結果 / 數據
- `python -m py_compile detect.py` 通過。
- 修正後，`Height Sensor` channel 讀取將被限制在 `25088 bytes`
  （`49` 行），不再延伸至 `DMTModulus` 等後續 channel 的資料區。
- `CLAUDE.md` 變更紀錄同步追加說明（根因、修正方式、後續注意事項）。

## 結論 / 討論
1. **根因確立**：`std.001` 推論異常的根本原因是 `read_nanoscope_
   channel()` 忽略 channel 自身的 `Data length` 邊界，導致部分掃描
   （中途中斷）檔案的高度資料被其他 channel（力學性質類，如
   `DMTModulus`、`Adhesion`）的數值污染，而非 AFM 物理模型或 Z 校正
   公式本身的問題。
2. 此修正具有**通用性**：任何因掃描中斷、回授失鎖等原因造成
   `Data length < n_px×n_lines×bpp` 的 Nanoscope 檔案，皆會受益於
   此邊界保護，不限於本次 `std.001` 案例。
3. **殘留限制**：即使修正讀取邊界，`std.001` 實際有效資料仍僅
   49/256 行（19%），resize 至 128×128 後垂直方向幾何將嚴重失真，
   修正後的預測結果僅可作為參考，不能視為可靠的去卷積結果。建議
   使用者確認此掃描是否需要重新量測，取得完整 256 行資料。
