# 電子研究紀錄簿

## 標題
`detect.py` Nanoscope 原始檔副檔名偵測邏輯放寬，支援描述性字尾

## 日期
2026-06-16

## 研究目的 / 背景
使用者以 `train14` 模型對真實掃描檔進行推論測試，將原始 Nanoscope
二進位檔 `std.000` 手動重新命名為 `std.000-after flatten`（內容未變，
僅檔名加註處理階段供自己辨識），執行：

```bash
python detect.py "...\flatten\std.000-after flatten" -m runs/train14/weights/AFM_MAE_autoencoder.keras
```

結果觸發例外：

```
PIL.UnidentifiedImageError: cannot identify image file
'...\std.000-after flatten'
```

需查明為何此檔案未被正確識別為 Nanoscope 格式。

## 方法 / 過程
1. 檢視 `detect.py` 中 `is_nanoscope_file()` 判斷邏輯：

   ```python
   def is_nanoscope_file(filepath):
       ext = os.path.splitext(filepath)[1]
       return bool(re.match(r'^\.\d{3}$', ext))
   ```

2. 以 Python `os.path.splitext()` 語意分析檔名 `std.000-after flatten`：
   檔名僅含一個 `.`，故 `splitext` 回傳的副檔名為整段
   `.000-after flatten`，而非單純的 `.000`。
3. 正則 `^\.\d{3}$` 要求**整段副檔名**恰好是 3 位數字結尾，
   `.000-after flatten` 不符合（多了描述文字），判斷邏輯回傳 `False`，
   程式因而誤判為一般圖片格式，改用 `PIL.Image.open()` 開檔，
   因內容是 Nanoscope 二進位格式而非圖片格式，PIL 解析失敗並丟出
   `UnidentifiedImageError`。
4. 向使用者確認該檔案實際格式：內容確認**仍為原始 Nanoscope 二進位
   結構**，僅檔名加註人工標記，非格式轉換。
5. 修正 `is_nanoscope_file()` 正則為 `^\.\d{3}`（移除結尾錨點
   `$`），只要求副檔名**以 3 位數字開頭**，允許其後接任意描述文字。

## 結果 / 數據
修正前後對照：

| 檔名 | 舊版判斷 | 新版判斷 |
|------|---------|---------|
| `std.000` | Nanoscope ✓ | Nanoscope ✓ |
| `std.004` | Nanoscope ✓ | Nanoscope ✓ |
| `std.000-after flatten` | 一般圖片 ✗（誤判） | Nanoscope ✓ |

`python -m py_compile detect.py` 通過，修正後可直接以原檔名執行推論，
不需使用者手動改檔名。

## 結論 / 討論
此為純粹的字串判斷邏輯瑕疵，與 AFM 物理或模型推論無關。修正後使用者
可自由在檔名後加註處理階段（如 `-after flatten`、`-corrected` 等）
作為個人標記，而不影響程式對 Nanoscope 格式的自動識別，降低操作摩擦。
